# Wizard создания персонажа: целевой уровень → создаём 1 уровня + XP

Документ для FE-разработчика. Меняется **семантика поля `level`** в визарде создания персонажа.
Бэкенд уже обновлён — фронт нужно привести в соответствие.

---

## 1. Что изменилось на бэкенде

Раньше: если в визарде указать уровень 5, персонаж создавался сразу 5 уровня
(`totalLevel = 5`, `classLevel = 5`, полный HP, `experience = 0`).

Теперь: персонаж **всегда создаётся 1 уровня**, а в `experience` кладётся порог XP для
запрошенного (целевого) уровня. Дальше игрок поднимает персонажа до целевого уровня через
обычную механику level-up, делая выбор фич/ASI/подкласса/заклинаний на каждом шаге.

Поле `level` в запросе `POST /api/campaigns/{campaignId}/characters` и
`POST /api/characters` (vanilla) теперь трактуется как **целевой** уровень (1–20), а не как
уровень, на котором персонаж появится.

### Пример

Запрос `level = 5` (Воин) приводит к созданию персонажа со следующим состоянием:

| Поле | Значение |
|---|---|
| `totalLevel` | `1` |
| `classLevel` | `1` |
| `experience` | `6500` (порог XP 5 уровня) |
| `hitDiceTotal` | `1d10` |
| `maxHp` / `currentHp` | как у 1 уровня |

С `experience = 6500` персонаж готов подняться 1→2→3→4→5 через level-up и **ни уровнем выше**
(порог 6 уровня — 14000).

## 2. Что нужно поменять во фронте (обязательно)

Поскольку персонаж создаётся 1 уровня, **визард должен собирать загрузку только для 1 уровня**,
а не для целевого. Иначе запрос будет отклонён валидацией.

### 2.1. Заклинания — только 1 уровня
- `cantripIds` — заговоры (level 0), как и раньше.
- `spellIds` — теперь принимаются только заклинания, доступные на **1 уровне** класса.
  Заклинание уровня выше 1 → бэкенд вернёт ошибку `400` («Spell (level X) exceeds max spell level …»).
- Количество известных заклинаний/заговоров показывайте по правилам **1 уровня** класса,
  независимо от выбранного целевого `level`.

### 2.2. `initialRewardSelections` — только группы наград 1 уровня
- Передавайте выборы только для reward-групп **1 уровня** класса (как и сейчас по факту).
- Выборы фич/ASI/подкласса/прочего для уровней 2+ через визард **не передаются** — игрок их
  сделает позже в level-up.

### 2.3. Прочие поля визарда
- `chosenSkillIds`, `abilityScores`, `scoreMethod`, `backgroundId`, `raceId`, `startingCoins` —
  без изменений.

## 3. Что показать пользователю (UX)

Рекомендуется явно объяснить игроку поведение, чтобы не было сюрприза:

- При выборе целевого уровня > 1 показать пометку вроде:
  «Персонаж будет создан 1 уровня и получит {XP} опыта — этого хватит, чтобы поднять его
  до {целевой уровень} через повышение уровня. Фичи, повышения характеристик, подкласс и
  заклинания старших уровней вы выберете в процессе повышения.»
- После создания персонаж в ответе придёт с `totalLevel = 1` и ненулевым `experience` —
  карточка/лист должны это корректно отображать (индикатор «готов к повышению уровня»
  обычно уже работает по `xpToNextLevel` из `GET /api/characters/{id}/level-up-options`).

---

## 4. Ответ создания (без изменений по структуре)

`ContentCharacterCreationResponse`:
```jsonc
{
  "id": "uuid",
  "name": "Hero",
  "classId": "uuid",
  "totalLevel": 1,          // ВСЕГДА 1 (раньше мог быть = целевому)
  "campaignId": "uuid|null",
  "skillProficiencyIds": ["uuid", ...],
  "knownSpellIds": ["uuid", ...]
}
```
Обратите внимание: `totalLevel` в ответе теперь всегда `1`. Если где-то на фронте
отображался ожидаемый уровень из этого поля — переключитесь на целевой `level` из формы
или на данные level-up.

---

## 5. Чек-лист для FE

- [ ] Поле `level` в форме переименовать/подписать как «целевой уровень» (значения 1–20).
- [ ] Списки заклинаний/заговоров и их лимиты — считать по 1 уровню класса.
- [ ] `initialRewardSelections` — собирать только для reward-групп 1 уровня.
- [ ] Не отправлять выборы старших уровней при создании.
- [ ] Показать пользователю пояснение про «создан 1 уровня + XP».
- [ ] Не полагаться на `totalLevel` из ответа создания как на целевой уровень (он всегда 1).

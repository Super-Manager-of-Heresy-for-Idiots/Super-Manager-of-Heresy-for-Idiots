# FE: cookie-сессия + бесшовное автопродление JWT

Дата генерации: 2026-06-19 16:14

## Что изменилось на бэкенде

Сессия теперь хранится в **HttpOnly-cookie**, а access-токен **бесшовно продлевается** через отдельный refresh-токен. Пользователь больше не «вылетает» через час — он остаётся залогинен, пока активен (по умолчанию окно бездействия 7 дней).

Введены два токена:

| Токен | Время жизни (по умолчанию) | Где лежит | Назначение |
|-------|----------------------------|-----------|------------|
| **access** | 1 час (`JWT_EXPIRATION_MS`) | HttpOnly-cookie `access_token` (path `/`) **и** в теле ответа | авторизует все REST/WS-запросы |
| **refresh** | 7 дней (`JWT_REFRESH_EXPIRATION_MS`) | HttpOnly-cookie `refresh_token` (path `/api/auth`) | только для продления сессии |

Обе cookie: `HttpOnly`, `SameSite=Lax`, `Secure=false` в dev / `true` в проде. JS **не может** прочитать токен из cookie — это сделано намеренно (защита от XSS).

## Контракт API

### POST /api/auth/login (без изменений в запросе)
Тело ответа прежнее:
```json
{ "success": true, "data": { "token": "<access>", "expiresIn": 3600000, "user": {...} }, "message": "Вход выполнен" }
```
**Дополнительно** ответ ставит `Set-Cookie: access_token=...` и `Set-Cookie: refresh_token=...`. Браузер сохранит их сам.

### POST /api/auth/refresh (НОВЫЙ)
- Запрос без тела. Браузер автоматически шлёт `refresh_token` cookie.
- Ответ — точно такой же формат, как у login (`token`, `expiresIn`, `user`) + обновлённые cookie (refresh-токен ротируется).
- Ошибки: `401` если refresh-токен невалиден/просрочен → нужно отправить пользователя на экран входа.

### POST /api/auth/logout (НОВЫЙ)
- Запрос без тела. Очищает обе cookie (`Max-Age=0`).
- Ответ: `{ "success": true, "data": null, "message": "Выход выполнен" }`.

## Что нужно сделать на фронте

1. **Включить отправку cookie на всех запросах к API.**
   - `fetch`: `credentials: 'include'`.
   - `axios`: `withCredentials: true` (глобально на инстансе).
   - Без этого cookie не будут уходить и сессия не заработает.

2. **Перестать хранить JWT в `localStorage`/`sessionStorage`.**
   - REST-авторизация теперь полностью на cookie — заголовок `Authorization` для REST больше не нужен (он всё ещё поддерживается как fallback, но использовать его не надо).
   - Access-токен из тела ответа держим **только в памяти** (нужен для WebSocket, см. п. 4).

3. **Бесшовное автопродление.** Реализовать любой из вариантов (рекомендуется комбинировать первый и второй):
   - **Реактивно:** на ответ `401` от любого защищённого запроса — один раз дёрнуть `POST /api/auth/refresh`, и при успехе повторить исходный запрос. При неудаче refresh → `logout` + редирект на вход. (Сделать защиту от бесконечного цикла: не рефрешить сам `/refresh`.)
   - **Проактивно:** по `expiresIn` из последнего ответа ставить таймер и вызывать `/refresh` за ~1 минуту до истечения access-токена, пока вкладка активна.
   - Несколько параллельных 401 должны триггерить **один** общий refresh (single-flight), остальные ждут его результата.

4. **WebSocket (STOMP).** WS-хендшейк по-прежнему авторизуется access-токеном в native-заголовке `Authorization: Bearer <token>` (cookie для STOMP-CONNECT не используется). Поэтому:
   - Брать `token` из тела ответа login/refresh и держать его в памяти.
   - После успешного `/refresh` **переподключать** WS с новым токеном (или обновлять заголовок CONNECT при реконнекте).

5. **Кнопка «Выйти».** Дёргать `POST /api/auth/logout` (он очистит cookie), затем чистить in-memory access-токен и состояние и редиректить на вход.

6. **Старт приложения / восстановление сессии.** При загрузке, если нет access-токена в памяти, попробовать `POST /api/auth/refresh`:
   - успех → пользователь залогинен (восстановили сессию из refresh-cookie, перезагрузка страницы больше не разлогинивает);
   - `401` → показать экран входа.

## Важно про окружения / CORS
- В dev фронт и API должны быть **одного origin** (через Vite-proxy `/api → :8080`) либо одного site — иначе `SameSite=Lax` cookie не уйдут. Текущая схема с прокси подходит.
- В проде по https бэкенд выставит `Secure=true`. Если фронт и API окажутся на **разных доменах**, потребуется `SameSite=None; Secure` (env `JWT_COOKIE_SAMESITE=None`, `JWT_COOKIE_SECURE=true`) и CORS с `allowCredentials`. Согласовать перед раскаткой.

## Чек-лист приёмки
- [ ] Логин ставит cookie `access_token` и `refresh_token`; запросы к API идут с cookie (`credentials: include`).
- [ ] JWT больше не лежит в localStorage.
- [ ] По истечении access-токена сессия молча продлевается, пользователь не разлогинивается.
- [ ] Перезагрузка вкладки сохраняет вход (восстановление через `/refresh`).
- [ ] «Выйти» вызывает `/api/auth/logout`, cookie очищаются, происходит редирект на вход.
- [ ] WS переподключается с новым токеном после refresh.
- [ ] Параллельные 401 вызывают один refresh, а не лавину.

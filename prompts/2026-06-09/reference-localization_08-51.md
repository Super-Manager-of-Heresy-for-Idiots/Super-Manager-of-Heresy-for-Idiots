# Локализация справочных данных (RU/EN) — инструкция для фронтенда

## Что изменилось на бэкенде

Справочные данные PHB (заклинания, расы, классы, предыстории, навыки владения,
валюты) теперь двуязычны. Каждое пользовательское текстовое поле хранится в БД в
двух версиях — английской (`*_engloc`) и русской (`*_rusloc`) — плюс каноничный
английский столбец как запасной вариант.

API **не** отдаёт обе версии. Вместо этого эндпоинт сам выбирает нужную по
параметру `lang` и возвращает обычные поля `name` / `description` / … уже на
выбранном языке. Структура ответа не изменилась — поменялось только содержимое
текстовых полей.

## Параметр `lang`

- Передаётся как **query-параметр**: `?lang=ru` или `?lang=en`.
- Допустимые значения: `ru`, `en`. Регистр не важен.
- **Необязательный.** Если не передан или значение нераспознано — используется `en`.

Рекомендация: добавить `lang` ко всем запросам справочников, подставляя текущую
локаль интерфейса пользователя (`i18n.language` → `'ru'` | `'en'`).

## Правило выбора значения (важно понимать поведение)

Для каждого текстового поля бэкенд возвращает:

1. версию запрошенного языка (`ru` → русскую, `en` → английскую);
2. если она пустая/`null` — версию **другого** языка;
3. если обе пустые — каноничный английский текст.

Итог: поле `name`/`description` **никогда не приходит `null`** из-за локализации.
Если перевода на русский ещё нет, при `lang=ru` придёт английский — это ожидаемое
поведение, отдельно обрабатывать на фронте не нужно.

## Затронутые эндпоинты

Параметр `lang` поддерживают оба набора справочников.

Vanilla (без кампании), `/api/reference/*`:

| Метод | Локализуемые поля |
|---|---|
| `GET /api/reference/classes?lang=ru` | `name`, `description`, `armorWeaponProficiencies`, вложенные `skillChoiceOptions[].name` |
| `GET /api/reference/races?lang=ru` | `name`, `description`, `traits[]`, `subraces[].name`, `subraces[].description` |
| `GET /api/reference/backgrounds?lang=ru` | `name`, `description`, `grantedExtras` |
| `GET /api/reference/skills?lang=ru` | `name` |
| `GET /api/reference/currencies?lang=ru` | `name` |
| `GET /api/reference/spells?lang=ru` | `name`, `description` |

Внутри кампании, `/api/campaigns/{campaignId}/reference/*` — те же шесть эндпоинтов
и те же поля, плюс существующие параметры (`classId`, `level`, `school` у spells).

`GET .../stat-types` **не** принимает `lang` (для характеристик переводов нет).

Параметр `lang` комбинируется с уже существующими фильтрами:
`GET /api/reference/spells?level=3&school=Evocation&lang=ru`.

## Примеры

Запрос:
```
GET /api/reference/spells?lang=ru
```
Ответ (сокращённо):
```json
{
  "success": true,
  "data": [
    {
      "id": "…",
      "name": "Огненный шар",
      "level": 3,
      "school": "Evocation",
      "description": "Яркая вспышка… 8d6 урона огнём при провале спасброска.",
      "availableToClassIds": ["…"]
    }
  ]
}
```

Тот же запрос с `?lang=en` (или без параметра) вернёт `"name": "Fireball"` и
английское описание. Поля `school`, `level`, `id` от языка не зависят.

## Что сделать на фронте

1. Прокинуть текущую локаль UI в `lang` для всех запросов к `/api/reference/*` и
   `/api/campaigns/{id}/reference/*` из таблицы выше.
2. Кэш справочников (если есть) ключевать **с учётом языка** — ответы для `ru` и
   `en` различаются. На бэкенде vanilla-эндпоинты тоже кэшируются по языку.
3. При переключении языка интерфейса — перезапросить/инвалидировать справочники.
4. Дополнительная обработка `null` из-за отсутствия перевода не требуется (см.
   правило выбора выше).

## Примечание про уже засеянные строки

Часть базовых записей (6 характеристик, 5 валют, 18 навыков, базовые расы/классы)
была залита более ранними миграциями. Их `*_rusloc`/`*_engloc` могут быть пустыми —
тогда по правилу фолбэка вернётся каноничный английский. Это не ошибка фронта; при
необходимости перевод этих строк до-заполняется отдельной миграцией на бэкенде.
